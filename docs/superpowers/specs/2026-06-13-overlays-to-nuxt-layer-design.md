# Frontend Overlays → Nuxt Layer Migration Design

## Summary

Migrate `frontend/overlays` (standalone Vue 3 SPA overlay renderer) into `web/layers/overlays/` as a Nuxt layer with CSR-only rendering via `routeRules`. This unifies the overlay system with the web application, eliminates the separate overlays build pipeline, and establishes `/o/:apiKey/*` as the canonical URL path with backward-compatible redirects from `/overlays/:apiKey/*`.

Additionally, move frontend workspace packages (`@twir/frontend-chat`, `@twir/frontend-faceit-stats`, `@twir/frontend-now-playing`, `@twir/fontsource`) from `libs/` into `web/lib/` as local directories — they're only consumed by web and no longer need to be workspace packages.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| SSR vs CSR only | `routeRules` with `{ ssr: false }` | Same pattern as dashboard layer — no `.client.vue` suffixes needed, cleaner |
| URL path | `/o/:apiKey/*` (canonical) | Shorter, cleaner URLs for OBS browser sources. `/overlays/:apiKey/*` redirects via `routeRules` |
| urql setup | Use web's `@bicou/nuxt-urql` | Single urql instance, subscription support already configured. Replace overlays' custom `plugins/urql.ts` |
| GraphQL codegen | Unified via web's pipeline | Web's codegen module scans `./layers/**/*.{ts,vue}` — overlay queries auto-discovered. No separate codegen, no `gql/` dir copy |
| Auth model | API key in URL params | No cookies, no auth middleware. API key passed via `api-key` header in urql fetchOptions |
| Frontend workspace packages | Move to `web/lib/` as plain folders | `@twir/frontend-chat`, `@twir/frontend-faceit-stats`, `@twir/frontend-now-playing`, `@twir/fontsource` only used by web — plain folders without `package.json`, import via `~~/lib/` |
| shadcn | Not used | Overlays don't use shadcn components — minimal UI, mostly canvas/Shadow DOM |
| Old frontend/overlays | Deprecate, remove from CI | Keep source temporarily for reference, remove from Docker build matrix |

### GraphQL Codegen

Web's `codegen.ts` already scans `./layers/**/*.{ts,vue}`. Overlay composables with `graphql()` calls will be auto-discovered. No separate codegen needed.

- Do NOT copy `frontend/overlays/src/gql/` — types generated by web's codegen into `app/gql/`
- All composables import from `~~/app/gql/graphql`
- `frontend/overlays/codegen.ts` becomes obsolete

### Data Flow

```
OBS Browser Source → /o/:apiKey/:overlayType
  → Nuxt page (CSR via routeRules, layout: false)
    → composable initializes urql with API key from route params
      → GraphQL subscriptions (real-time data)
      → REST calls (initial data)
        → Render overlay (canvas/Shadow DOM/DOM)
```

## Key Constraints

### API Key Authentication

Overlays use API key in URL params, not cookies. The urql client must:
- Extract `apiKey` from route params (`useRoute().params.apiKey`)
- Pass it as `api-key` and `x-api-key` headers in fetchOptions
- Pass it as `api-key` in WebSocket connection params for subscriptions

This replaces the custom `plugins/urql.ts` from frontend/overlays with a Nuxt-compatible composable.

### Client-Only Rendering

All overlay pages use `routeRules` approach (same as dashboard):
- Layer `nuxt.config.ts` sets `/o/:apiKey/**` → `{ ssr: false }`
- Pages use `definePageMeta({ layout: false })` — no layout wrapper
- No `.client.vue` suffixes needed
- Browser-only APIs (Canvas, Web Audio, Shadow DOM, WebSocket) work because SSR is disabled for these routes

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| TMI.js SSR issues | `routeRules` with `{ ssr: false }` — same proven pattern as dashboard |
| OBS WebSocket in browser | Already browser-compatible, no changes needed |
| GraphQL subscription setup | Web's urql already has subscriptionExchange configured |
| Local lib imports | Use `~~/lib/` prefix (Nuxt alias) — consistent with existing dashboard patterns |
| Redirect performance | Nuxt routeRules redirect is server-side (301), fast |
| API key in URL (security) | Same as current — API key is designed for browser-source URLs |

## Migration Strategy

Incremental — each overlay type can be migrated and tested independently:

1. **Workspace packages** — Move libs to `web/lib/`, update imports across web
2. **Infrastructure** — Layer config, dependencies, redirect rules
3. **Shared code** — Composables, components, helpers, types
4. **GraphQL** — Queries, codegen, urql adaptation
5. **Pages** — One overlay type at a time
6. **Dashboard links** — Update URL generation
7. **Cleanup** — Deprecate old app, remove from CI

## Success Criteria

- All 10 overlay types render correctly at `/o/:apiKey/:type`
- `/overlays/:apiKey/:type` redirects to `/o/:apiKey/:type`
- Dashboard "copy overlay link" generates `/o/` URLs
- GraphQL subscriptions work for real-time overlays
- TMI.js chat overlay connects and displays messages
- OBS WebSocket controller connects to OBS
- `frontend/overlays` removed from CI build matrix
- No regression in existing `/o/:apiKey/valorant-stats`
- `@twir/frontend-chat`, `@twir/frontend-faceit-stats`, `@twir/frontend-now-playing`, `@twir/fontsource` removed from workspace, used as plain folders in `web/lib/`
